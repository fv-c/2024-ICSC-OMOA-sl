Foreword

About the Reverberators project (1)

The Reverberators project was born in the educational context of the Conservatorio di Musica “N. Piccinni” of Bari (IT).

The starting point was identified in Manfred Schroeder's literary works on the creation of artificial reverberations with qualities similar to natural ones. [1, 2]

About the Reverberators project (2)

Those works reports reverberator's block diagrams and impulse response (IR).

The project was therefore a path of reconstruction and conscious implementation [3] of historical reverberators, with a strong view to their creative expansion.

The chosen language was Faust. [4]

"reconstruction and conscious implementation" are the characteristics that we have related to the idea of ​​"open architecture", in the sense of "open to understanding", but also of shareability.

Why Faust (Functional Audio Stream) initially?

  • text-based programming language
  • highest level functions are implemented using the same language
    (up to the basic “primitive” components)
  • it has tools to analyze IR sample-by-sample and visualize block diagrams

Why analysis tools?

Faust's analytical ability had allowed us to adopt a series of compensation strategies for the delay samples that did not emerge from the block diagram reported in the literature.
This was possible by analyzing the IR of our processors, comparing it with the one present in the papers, making the analytical tool essential for an in-depth understanding of the reconstruction process.

Samplerate implementations of these components allowed us to face a problem that would otherwise be impossible to discover with buffersize languages, without any sample-by-sample analitical structure. The feedback loops implementations require at least one sample to execute, feature not mentioned by Schroeder due to hardware differences with his machines.

Educational motivations

Csound has many affinities with Faust: it is a text-based system, which allows for careful analysis of the flow of operations on samples; furthermore the definition of functions (User Defined Opcodes) can be done using the csound language itself.
Another detail is that the Csound language has long been an integral part of our school's teaching.

Creating the IR analysis tool (Csound+Wolframscript)

						
							instr 1
							setksmps 1
							ires system_i 1, "mkdir -p PLOT"
							iT = 1;time
							iG = 1/ sqrt(2);gain
							iSR = sr;sample rate
							aDirac mpulse 1,1
							aR = 0
							iLenTable = 32
							giFDL_PLOT ftgen 2, 0, iLenTable, -2, 0
							aFDL_PLOT = FDL_SCH(aDirac, iT, iG, iSR);process to test
							aIndex phasor sr/iLenTable
							tablew aFDL_PLOT, aIndex * iLenTable, giFDL_PLOT
							schedule 2, iLenTable/sr, 1
						  endin
						  instr 2
							ftprint giFDL_PLOT
							ftsave "PLOT/1FDL_PLOT.txt", 1, giFDL_PLOT
							ires system_i 1, "wolframscript -script plot.wls"
							exitnow
						  endin
In the example a user defined opcode (FDL_SCH) is tested, which is a basic comb filter rebuilt on the Schroeder model. Of course csound also has an opcode (comb) but the aim was to compare the Schroeder one with the opcodes already available (not directly inspectable, because compiled).

Creative opening: par & seq

						
				;--------PAR FDL--------------------------------------------------
				opcode PAR_FDL_SCH, a, ai[]i[]iii
					aPulse, iTfdl[], iGfdl[], iSR, iN, icnt  xin
					icnt  =  icnt + 1
					aFDL = FDL_SCH(aPulse, iTfdl[icnt - 1], iGfdl[icnt - 1], iSR)
					aMix init 0
					if icnt < iN then
						aMix PAR_FDL_SCH aPulse, iTfdl, iGfdl, iSR, iN, icnt
					endif
					xout aMix + aFDL
				endop
				;--------SEQ APF-------------------------------------------------
				opcode SEQ_APF_SCH, a, ai[]i[]iii
					aPulse, iTapf[], iGapf[], iSR, iN, icnt  xin
					icnt  =  icnt + 1
					aAPF = APF_SCH(aPulse , iTapf[icnt - 1], iGapf[icnt - 1], iSR)
					if icnt < iN then
						aAPF SEQ_APF_SCH aAPF, iTapf, iGapf, iSR, iN, icnt
					endif
					xout aAPF
				endop
Usually csound is used for the development of audio dsp circuits, but it does not natively have functions that allow to "emulate" the basic syntax for the direct composition of units. Specifically, there are no syntactic constructs dedicated to the composition in series or in parallel of components. If it is easy to create chains of a few components with the current syntax, it is not as easy to create longer and more complex compositions of circuits. The original reverberators project explored precisely the creative expansion of historical circuits, with a view to exploring the sonic potential of circuits with a high number of components in series and in parallel.

Future directions

From the work we can extrapolate two concepts:

  1. The measurability of the system (analysis).
  2. The variability of response of the system if we enter input values ​​reasoned on the basis of the analytical step

Through these two principles we can build more or less complex tools, which are not only "effect" but also a playable instrument.
